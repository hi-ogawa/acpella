# Per-Message Runtime Metadata

## Problem Context And Approach

Issue: <https://github.com/hi-ogawa/acpella/issues/28>

acpella sends Telegram text directly to the ACP agent. The agent sees session-level instructions, but it does not reliably see the receive time and source metadata for each incoming message. That makes relative time references like "today", "now", or elapsed-time checks depend on the model running a tool such as `date`, which measures tool execution time rather than message receive time.

Inject a small trusted metadata block at runtime for every user prompt sent to the ACP agent. Keep it per-message and separate from the durable custom prompt file. The first prompt may still include `.acpella/AGENTS.md`, but the metadata should be generated by `handler.ts` from the current grammY context and added around the actual user text for both new and resumed sessions.

## OpenClaw Reference

Local reference repo: `~/code/others/openclaw`

- `extensions/telegram/src/bot-handlers.runtime.ts`
  - Captures `receivedAtMs: Date.now()` at Telegram ingress and carries it through debounce/buffered message processing.
- `extensions/telegram/src/bot-message-context.types.ts`
  - Defines `TelegramMessageContextOptions.receivedAtMs`.
- `extensions/telegram/src/bot-message-context.session.ts`
  - Builds the inbound context payload from platform fields: `Surface`, `Provider`, `ChatType`, `MessageSid`, `Timestamp`, `SenderId`, `OriginatingTo`, and related metadata.
- `src/auto-reply/reply/inbound-meta.ts`
  - Separates trusted system metadata from user-role conversation metadata.
  - Keeps attacker-controlled values such as names/history out of trusted metadata.
- `src/auto-reply/envelope.ts`
  - Formats message timestamps centrally and includes timezone-aware text when configured.

OpenClaw's main lesson for acpella: capture receive time at ingress, carry it as structured runtime data, and keep the metadata formatter centralized so tests can pin the exact prompt shape.

## acpella Reference Files

- `src/cli.ts`
  - Receives grammY `message:text` updates, computes the Telegram session name, and calls `handler.handle(...)`.
- `src/handler.ts`
  - Owns local command routing and sends normal prompts to `session.prompt(...)`.
  - Current `formatFirstPrompt(...)` is the only prompt wrapper.
- `src/acp/index.ts`
  - Sends ACP prompts as `prompt: [{ type: "text", text }]`.
- `src/lib/telegram.ts`
  - Contains Telegram session naming helpers and can host small Telegram metadata helpers if they stay surface-specific.
- `src/repl.ts`
  - Creates fake Telegram updates with `message.date`, useful for local/e2e metadata coverage.

## Implementation Plan

1. Add a small `MessageRuntimeMetadata` type with fields:
   - `received_at`
   - `timezone`
   - `surface`
   - `chat_type`
   - `chat_id`
   - `message_id`
   - optional `sender_id`
2. Capture `received_at` as early as practical in `src/cli.ts` before allowlist checks call into the handler. Pass it into `handler.handle(...)`.
3. Derive metadata in `src/handler.ts` from the passed receive time and `Context`:
   - `received_at`: ISO 8601 with local offset, not plain UTC `Z`.
   - `timezone`: `Intl.DateTimeFormat().resolvedOptions().timeZone` when available, otherwise `"UTC"`.
   - `surface`: `"telegram"` for real Telegram and REPL, since REPL uses Telegram-shaped updates.
   - `chat_type`: map Telegram `private` to `dm`, otherwise `group`.
   - `chat_id`, `message_id`, `sender_id`: stringify IDs to avoid precision surprises in prompt text.
4. Add a central formatter, e.g. `formatPromptWithMessageMetadata({ metadata, userText })`, that emits:
   - a concise `<message_metadata>` block
   - only generated metadata fields
   - the original user text after the metadata block

## Scope Notes

- Do not add configuration for timezone in the first pass; use runtime/user local timezone.
- Do not include sender names, chat titles, reply snippets, or history in trusted metadata.
- Do not change ACP session storage or agent lifecycle.
- Avoid broad OpenClaw-style prompt-cache/system-prompt separation for now. A small per-prompt metadata block is enough for the issue.
