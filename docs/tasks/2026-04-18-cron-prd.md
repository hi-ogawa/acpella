# First-Class Cron PRD

## Problem Context

Issue context: <https://github.com/hi-ogawa/acpella/issues/30>

acpella should support scheduled agent work as a first-class product capability. Today, recurring workflows depend on external cron glue, stale OpenClaw assumptions, or manual prompts. That makes scheduled work hard to inspect, hard to modify from chat, and easy to break when the runtime shape changes.

The old issue is useful background, but it should not be treated as the implementation plan. This PRD defines the minimum product behavior needed to iterate toward cron support without preserving stale structure.

## Product Goal

Allow the user to register a recurring system-level scheduled instruction that acpella runs at a chosen time, sends to the correct agent conversation, and delivers the result back through the existing messaging surface.

The MVP should make scheduled work feel like a native acpella feature: visible, controllable, and predictable from the user's perspective.

## MVP User Experience

The first version starts with explicit acpella commands instead of asking the agent to create or modify schedules. A cron is attached to the current chat/session where the command is issued.

```text
/cron add morning-planning 0 8 * * 1-5 Asia/Tokyo Run my daily planning prompt.
```

After the cron is created, the user can use system-level cron commands to inspect what exists, temporarily stop it, resume it, or delete it.

When a scheduled job fires, the resulting assistant message should appear as a proactive message in the target chat rather than as a reply to an old user message.

## MVP Scope

The first version should support this command surface:

```text
/cron add <id> <minute> <hour> <day-of-month> <month> <day-of-week> <timezone> <prompt...>
/cron list
/cron show <id>
/cron enable <id>
/cron disable <id>
/cron delete <id>
```

### Command Semantics

`/cron add` creates an enabled recurring job with the provided user-facing id. The id must match `[a-zA-Z0-9_-]+`. The five schedule fields use standard cron field order. The timezone is required. The prompt is the remaining text after the timezone. The target chat and target agent conversation are the current acpella session where the command was issued.

`/cron list` shows all cron jobs known to acpella with id, enabled state, schedule, timezone, target session, next run if known, and last result if known.

`/cron show <id>` shows one job with its full prompt and status.

`/cron enable <id>` and `/cron disable <id>` toggle whether the job can fire.

`/cron delete <id>` removes the job. The MVP can require the exact id as sufficient intent; no separate confirmation is required.

## Architecture Concept

Cron should be modeled as a small acpella-owned subsystem beside the existing session state, not as agent memory and not as Telegram-specific command text hidden in prompts.

The core architecture has four concepts:

- Cron definitions: durable user intent.
- Cron run state: operational bookkeeping about what fired and what happened.
- Scheduler loop: in-process service responsibility that decides when a job is due.
- Trigger executor: shared path that turns a due job into an agent prompt and proactive delivery.

### Persistence Model

Persist cron definitions separately from existing session state under `.acpella/cron.json`.

The file stores definitions under `jobs[id]` and includes `version: 1`.

Each definition contains:

- `id`
- `enabled`
- `schedule`: five cron fields as one string
- `timezone`
- `prompt`
- `target.sessionName`
- `target.telegram.chatId`
- `target.telegram.messageThreadId` when the target is a Telegram forum topic

Example definition:

```json
{
  "version": 1,
  "jobs": {
    "morning-planning": {
      "id": "morning-planning",
      "enabled": true,
      "schedule": "0 8 * * 1-5",
      "timezone": "Asia/Tokyo",
      "prompt": "Run my daily planning prompt.",
      "target": {
        "sessionName": "telegram:12345",
        "telegram": {
          "chatId": 12345
        }
      }
    }
  }
}
```

Persist run state separately from definitions under `.acpella/cron-state.json`.

Run state includes `version: 1` and stores runs under `runs[cronId][scheduledAt]`. `scheduledAt`, `startedAt`, and `finishedAt` are stored as canonical UTC instant strings.

Each run contains:

- `id`: UUID
- `scheduledAt`: the scheduled occurrence instant from the cron expression
- `startedAt`
- `finishedAt` when the run has completed
- `status`: `running`, `succeeded`, or `failed`
- `error` when failed

Example run state:

```json
{
  "version": 1,
  "runs": {
    "morning-planning": {
      "2026-04-19T23:00:00Z": {
        "id": "542b7f1a-83f4-4927-bc5a-53dcb48924e6",
        "scheduledAt": "2026-04-19T23:00:00Z",
        "startedAt": "2026-04-19T23:00:04Z",
        "finishedAt": "2026-04-19T23:00:42Z",
        "status": "succeeded"
      }
    }
  }
}
```

Before starting a scheduled run, acpella checks whether `runs[cronId][scheduledAt]` already exists. If it exists, acpella does not start another scheduled run for that occurrence.

When starting a scheduled run, acpella writes the run record with `status: "running"` before sending the prompt to the agent.

`/cron list` and `/cron show <id>` read definitions and the latest relevant runs to display last result and failure information.

Early versions may make breaking JSON shape changes without migration. The `version` field is present so acpella can fail clearly or add migration later if the files outlive the MVP shape.

### Scheduler Ownership

The scheduler belongs to the acpella service process. It starts with the service, loads cron definitions, computes due jobs, and periodically wakes to trigger work.

For the MVP, a single running acpella process is the expected owner. The scheduler does not need multi-process coordination or distributed locking.

The scheduler ignores disabled jobs. For enabled jobs, it computes the next due scheduled occurrence and asks the trigger executor to start that occurrence.

Minute-level scheduling is sufficient for the MVP; second-level precision is not required.

### Trigger Flow

Scheduled execution should reuse the same conceptual route as a user prompt:

```text
due cron occurrence
  -> load cron definition and target session
  -> build a prompt with cron trigger metadata
  -> send prompt to the target agent conversation
  -> deliver assistant output to the target Telegram chat as a proactive message
  -> record run status
```

The prompt sent to the agent should include explicit trigger metadata before the user-authored prompt text. The metadata should identify at least:

- trigger type: cron
- cron id
- scheduled occurrence time
- actual fired time
- timezone
- target session name

Run identity is stored as a UTC instant. Prompt metadata can render the scheduled occurrence in the job timezone so the agent sees the local scheduled time.

Example prompt:

```xml
<trigger_metadata>
trigger: cron
cron_id: morning-planning
scheduled_at: 2026-04-20T08:00:00+09:00
started_at: 2026-04-20T08:00:04+09:00
timezone: Asia/Tokyo
session_name: telegram:12345
</trigger_metadata>

Run my daily planning prompt.
```

This lets the agent distinguish scheduled work from live user messages without requiring cron-specific memory inside the agent.

### Command Boundary

`/cron` commands should be handled as acpella system commands. They should validate user input, persist definitions, and inspect status without forwarding command text to the agent.

`/cron add` captures the current Telegram session as the target. That means the command layer needs access to both the acpella session name and the Telegram delivery address. If a surface cannot provide proactive delivery details, cron creation should fail with a clear system message.

`target.sessionName` is the acpella session key used to find the existing ACP agent conversation. `target.telegram` is the outbound delivery address used after the agent responds. They are both captured from the same inbound Telegram context, but they serve different sides of the cron run.

For MVP, cron definitions store explicit Telegram delivery fields under `target.telegram`. This keeps `.acpella/cron.json` readable and validates the only supported delivery surface. The cron executor should still receive Telegram sending as an injected function rather than importing grammY or calling Telegram APIs directly.

### Delivery Boundary

Cron execution should not depend on a live inbound Telegram message. The stored target must contain enough delivery information for acpella to send a new bot message later.

Scheduled output should not use `reply_to_message_id`. It should be sent as a normal bot message to the stored chat/thread target. For forum topics, the target should preserve the thread/topic identifier so proactive messages land in the same conversation context as the original `/cron add`.

### Failure And Restart Model

The MVP should record failures in run state and expose them through `/cron list` and `/cron show`. A failed scheduled run should not retry in a tight loop or spam the target chat.

On startup, acpella loads definitions and run state before scheduling jobs. It does not replay old missed scheduled occurrences in the MVP. The scheduler starts from the next occurrence after startup time.

## Implementation Sketch

Keep cron code in a small dedicated area and connect it to existing boundaries instead of spreading scheduling logic through `cli.ts` and `handler.ts`.

The split should keep acpella policy separate from generic cron mechanics. Store and executor code are acpella-owned: they know about `.acpella` files, ACP sessions, run records, and delivery. Schedule calculation should stay acpella-independent: it takes a cron expression, timezone, and reference time, then returns occurrence times. That makes it possible to wrap a package or replace the implementation later without changing cron persistence or execution policy.

Likely new files:

- `src/cron/store.ts`
  - Owns zod schemas for `.acpella/cron.json` and `.acpella/cron-state.json`.
  - Reads/writes cron definitions and run state.
  - Provides operations such as add, list, show, enable, disable, delete, start run, finish run.
  - Uses `timer.ts` validation helpers for schedule/timezone fields instead of duplicating cron validation.
- `src/cron/executor.ts`
  - Builds cron trigger metadata.
  - Starts a run record before sending the prompt.
  - Calls the shared agent prompt path and records success/failure.
- `src/cron/timer.ts`
  - Generic cron timer engine.
  - Validates five-field cron expressions and computes timezone-aware occurrences.
  - Exposes validation helpers separately from timer lifecycle APIs so persistence and commands can validate schedules without starting timers.
  - Owns timer/callback mechanics for due occurrences.
  - Exposes generic entries and callbacks such as `{ id, schedule, timezone }` and `onDue({ id, scheduledAt })`.
  - Uses minute-level precision.
  - Should not know about `.acpella`, ACP sessions, Telegram, run records, or delivery.
- `src/cron/runner.ts`
  - Acpella cron runner around `timer.ts`.
  - Loads enabled cron definitions from the store.
  - Registers them with the generic timer.
  - On due callbacks, checks run state and calls the executor.
  - Rebuilds or updates scheduled entries after `/cron` command changes.
  - Starts from the next occurrence after service startup.

Likely existing file changes:

- `src/config.ts`
  - Add `cronFile` and `cronStateFile` paths under `.acpella`.
- `src/cli.ts`
  - Pass Telegram delivery context into `handler.handle`.
  - Start the cron scheduler after Telegram bot setup.
  - Provide a proactive Telegram send function that sends normal messages without `reply_to_message_id`.
- `src/handler.ts`
  - Add `/cron` system commands.
  - Expose or factor the prompt execution path so cron execution can send a prompt to an existing session without going through inbound command text.
  - Accept enough delivery context on inbound Telegram messages for `/cron add` to persist `chatId` and optional `messageThreadId`.
    `src/lib/prompt.ts`
  - Add a helper for cron trigger metadata if it fits beside existing message metadata helpers.

Testing shape:

- Unit tests for `src/cron/store.ts`
  - Initializes empty files.
  - Validates `version: 1`.
  - Adds, enables, disables, deletes, and persists jobs.
  - Writes `running`, `succeeded`, and `failed` run records.
  - Refuses to start a run when `runs[cronId][scheduledAt]` already exists.
- Unit tests for `src/cron/timer.ts`
  - Accepts valid five-field schedules.
  - Rejects malformed schedules.
  - Computes next occurrence across the configured timezone.
- Unit tests for `/cron` command handling in `src/handler.test.ts`
  - `/cron add` persists a job bound to the current Telegram target.
  - `/cron list`, `/cron show`, `/cron enable`, `/cron disable`, and `/cron delete` render useful system replies.
  - `/cron add` fails clearly when the inbound context has no proactive Telegram delivery target.
- Unit tests for trigger prompt construction
  - Cron metadata appears before the stored prompt.
  - Metadata includes cron id, `scheduled_at`, `started_at`, timezone, and session name.
- Runner tests with fake time or direct due-occurrence calls
  - Disabled jobs do not run.
  - Enabled due jobs call the executor once.
  - Existing `runs[cronId][scheduledAt]` prevents a duplicate run.
- Minimal integration coverage
  - REPL does not need full cron scheduling for MVP.
  - Telegram proactive delivery can be covered by injecting a fake send function rather than hitting the real Telegram API.

## Follow-Ups

These are intentionally outside the MVP and can be handled after the first command-driven cron path works.

- Correctness hardening
  - Handling for overlapping runs when the same job is still active at its next scheduled time.
  - Better missed-run policy controls.
  - Recovery or cleanup for stale `running` run records after a process crash.
  - Retention or truncation for old cron run records.
  - Multi-process coordination or distributed locking.
- Command UX
  - Flag-based command parsing, quoted arguments, or rich command forms.
  - Human-readable names separate from ids.
  - Manual trigger command such as `/cron run <id>`.
  - Richer schedule previews.
- Scheduling features
  - One-off scheduled jobs.
  - Pause until date/time.
  - Quiet hours or urgency levels.
  - Holiday or business-day calendar semantics.
  - Per-job history.
- Delivery and safety
  - Agent reply convention such as `NO_REPLY` to suppress outbound delivery for cron prompts that intentionally decide not to send anything.
  - Proactive failure notifications outside `/cron list` and `/cron show`.
  - Group-chat safety rules for proactive scheduled messages.
  - Multiple recipients or fan-out delivery.
- Product expansion
  - Natural-language cron creation through the agent.
  - Agent-side cron creation/update tools with explicit user approval.
  - Multiple messaging surfaces.
  - Import or recreation helpers for existing OpenClaw-style scheduled workflows.
  - Visual editor or dashboard.
