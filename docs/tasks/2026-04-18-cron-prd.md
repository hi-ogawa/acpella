# First-Class Cron PRD

## Problem Context

Issue context: <https://github.com/hi-ogawa/acpella/issues/30>

acpella should support scheduled agent work as a first-class product capability. Today, recurring workflows depend on external cron glue, stale OpenClaw assumptions, or manual prompts. That makes scheduled work hard to inspect, hard to modify from chat, and easy to break when the runtime shape changes.

The old issue is useful background, but it should not be treated as the implementation plan. This PRD defines the minimum product behavior needed to iterate toward cron support without preserving stale structure.

## Product Goal

Allow the user to register a recurring system-level scheduled instruction that acpella runs at a chosen time, sends to the correct agent conversation, and delivers the result back through the existing messaging surface.

The MVP should make scheduled work feel like a native acpella feature: visible, controllable, and predictable from the user's perspective.

## MVP User Experience

The first version starts with explicit acpella commands instead of asking the agent to create or modify schedules. A cron is attached to the current chat/session where the command is issued.

```text
/cron add morning-planning 0 8 * * 1-5 Asia/Tokyo Run my daily planning prompt.
```

After the cron is created, the user can use system-level cron commands to inspect what exists, temporarily stop it, resume it, delete it, or trigger it manually for testing.

When a scheduled job fires, the resulting assistant message should appear as a proactive message in the target chat rather than as a reply to an old user message.

## MVP Scope

The first version should support this command surface:

```text
/cron add <id> <minute> <hour> <day-of-month> <month> <day-of-week> <timezone> <prompt...>
/cron list
/cron show <id>
/cron enable <id>
/cron disable <id>
/cron delete <id>
/cron run <id>
```

### Command Semantics

`/cron add` creates an enabled recurring job. The five schedule fields use standard cron field order. The timezone is required. The prompt is the remaining text after the timezone. The target chat and target agent conversation are the current acpella session where the command was issued.

`/cron list` shows all cron jobs known to acpella with id, enabled state, schedule, timezone, target session, next run if known, and last result if known.

`/cron show <id>` shows one job with its full prompt and status.

`/cron enable <id>` and `/cron disable <id>` toggle whether the job can fire.

`/cron delete <id>` removes the job. The MVP can require the exact id as sufficient intent; no separate confirmation is required.

`/cron run <id>` manually triggers the job using the same prompt, target session, and delivery path as a scheduled run.

## Architecture Concept

Cron should be modeled as a small acpella-owned subsystem beside the existing session state, not as agent memory and not as Telegram-specific command text hidden in prompts.

The core architecture has four concepts:

- Cron definitions: durable user intent.
- Cron run state: operational bookkeeping about what fired and what happened.
- Scheduler loop: in-process service responsibility that decides when a job is due.
- Trigger executor: shared path that turns a due job into an agent prompt and proactive delivery.

### Persistence Model

Persist cron definitions separately from existing session state under `.acpella/cron.json`.

Each definition contains:

- `id`
- `enabled`
- `schedule`: five cron fields
- `timezone`
- `prompt`
- `target.sessionName`
- `target.surface`: `telegram` for MVP
- `target.telegram.chatId`
- `target.telegram.messageThreadId` when the target is a Telegram forum topic
- `createdAt`
- `updatedAt`

Persist run state separately from definitions under `.acpella/cron-state.json`.

Run state contains attempt records. Each attempt contains:

- `id`
- `cronId`
- `source`: `scheduled` or `manual`
- `scheduledFor` for scheduled attempts
- `startedAt`
- `finishedAt` when the attempt has completed
- `status`: `running`, `succeeded`, `failed`, or `skipped`
- `error` when failed

For scheduled attempts, `(cronId, scheduledFor)` is unique. Before starting a scheduled run, acpella checks whether an attempt already exists for the same `cronId` and `scheduledFor`. If one exists, acpella does not start another scheduled run for that occurrence.

Manual `/cron run <id>` creates a manual attempt. Manual attempts are not keyed by `scheduledFor`.

`/cron list` and `/cron show <id>` read definitions and the latest relevant attempts to display last result and failure information.

### Scheduler Ownership

The scheduler belongs to the acpella service process. It starts with the service, loads cron definitions, computes due jobs, and periodically wakes to trigger work.

For the MVP, a single running acpella process is the expected owner. The scheduler does not need multi-process coordination or distributed locking.

The scheduler ignores disabled jobs. For enabled jobs, it computes the next due scheduled occurrence and asks the trigger executor to start that occurrence.

### Trigger Flow

Scheduled execution should reuse the same conceptual route as a user prompt:

```text
due cron occurrence
  -> load cron definition and target session
  -> build a prompt with cron trigger metadata
  -> send prompt to the target agent conversation
  -> deliver assistant output to the target Telegram chat as a proactive message
  -> record run status
```

The prompt sent to the agent should include explicit trigger metadata before the user-authored prompt text. The metadata should identify at least:

- trigger type: cron
- cron id
- scheduled occurrence time
- actual fired time
- timezone
- target session name

This lets the agent distinguish scheduled work from live user messages without requiring cron-specific memory inside the agent.

### Command Boundary

`/cron` commands should be handled as acpella system commands. They should validate user input, persist definitions, inspect status, and trigger manual runs without forwarding command text to the agent.

`/cron add` captures the current Telegram session as the target. That means the command layer needs access to both the acpella session name and the Telegram delivery address. If a surface cannot provide proactive delivery details, cron creation should fail with a clear system message.

Manual `/cron run <id>` should go through the same trigger executor as a scheduled run. The main difference is that its occurrence source is manual rather than scheduler-driven, which should be reflected in run status or metadata if useful.

### Delivery Boundary

Cron execution should not depend on a live inbound Telegram message. The stored target must contain enough delivery information for acpella to send a new bot message later.

Scheduled output should not use `reply_to_message_id`. It should be sent as a normal bot message to the stored chat/thread target. For forum topics, the target should preserve the thread/topic identifier so proactive messages land in the same conversation context as the original `/cron add`.

### Failure And Restart Model

The MVP should record failures in run state and expose them through `/cron list` and `/cron show`. A failed scheduled run should not retry in a tight loop or spam the target chat.

On startup, acpella loads definitions and run state before scheduling jobs. It does not replay old missed scheduled occurrences in the MVP. The scheduler starts from the next occurrence after startup time.

## Follow-Ups

These are intentionally outside the MVP and can be handled after the first command-driven cron path works.

- Correctness hardening
  - Handling for overlapping runs when the same job is still active at its next scheduled time.
  - Better missed-run policy controls.
  - Multi-process coordination or distributed locking.
- Command UX
  - Flag-based command parsing, quoted arguments, or rich command forms.
  - Human-readable names separate from ids.
  - Richer schedule previews.
- Scheduling features
  - One-off scheduled jobs.
  - Pause until date/time.
  - Quiet hours or urgency levels.
  - Holiday or business-day calendar semantics.
  - Per-job history.
- Delivery and safety
  - Proactive failure notifications outside `/cron list` and `/cron show`.
  - Group-chat safety rules for proactive scheduled messages.
  - Multiple recipients or fan-out delivery.
- Product expansion
  - Natural-language cron creation through the agent.
  - Agent-side cron creation/update tools with explicit user approval.
  - Multiple messaging surfaces.
  - Import or recreation helpers for existing OpenClaw-style scheduled workflows.
  - Visual editor or dashboard.
